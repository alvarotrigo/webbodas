## Key Information Documented

### Access Token (`POLAR_ACCESS_TOKEN`)

**What It Is:**
- API access token for making authenticated requests TO Polar
- Format: `polar_at_xxxxxxxxxxxxx`
- Retrieved from: Polar Dashboard → Settings → API → Access Tokens

**Required Scopes (ONLY these 4):**
1. `customers:read` - Read customer information
2. `orders:read` - Read one-time orders (lifetime deals)
3. `checkouts:read` - **Critical:** Verify payments in success redirect
4. `subscriptions:read` - Read subscription data

**Why Only Read Scopes:**
- Application only reads data from Polar
- Never creates or modifies Polar data
- Write permissions would be security risk (principle of least privilege)

**Usage:**
- Used by `api/polar-success.php`
- Calls `getPolarCheckout($checkoutId)` to verify payment
- Prevents fake success URL manipulation

**Expiration:**
- Expires after 90-365 days (user's choice)
- Must be manually regenerated
- Code does NOT automatically refresh
- Set calendar reminder before expiration
- Webhooks continue working when token expires

**Renewal Process:**
1. Create new token with same 4 scopes
2. Update `.env` file
3. Restart server
4. Optional: Revoke old token

### Webhook Secret (`POLAR_WEBHOOK_SECRET`)

**What It Is:**
- Secret key for HMAC-SHA256 signature verification
- Format: `polar_whs_xxxxxxxxxxxxx`
- Retrieved from: Polar Dashboard → Settings → Webhooks → Create Webhook

**Important Notes:**
- **Generated by Polar** (not by you)
- Shown once when creating webhook
- Does NOT expire
- Different from access token

**Usage:**
- Used by `api/polar-webhook.php`
- Calls `verifyPolarWebhookSignature($payload, $signature)`
- Prevents fake webhook requests

**Why Needed:**
- Verifies webhook POST requests are actually from Polar
- Prevents attackers from granting themselves free access
- Works independently from access token

### Why Both Are Required

**Access Token Security:**
- Verifies payment actually succeeded via API call
- Prevents fake success URL redirects
- Used when user returns to your site

**Webhook Secret Security:**
- Verifies webhook request is actually from Polar
- Prevents fake webhook POST requests
- Used for ongoing subscription events

**Together:**
- Two independent verification methods
- Redundant payment confirmation
- Handle different attack vectors

## Security Enhancements Documented

1. **Principle of Least Privilege:**
   - Only 4 read scopes required
   - No write permissions needed
   - Reduces attack surface

2. **Token Rotation:**
   - Recommendation for 90-day expiration
   - Calendar reminder setup
   - Zero-downtime rotation process

3. **Dual Verification:**
   - Access token for API verification
   - Webhook secret for signature verification
   - Independent security layers

4. **Best Practices:**
   - Never commit secrets to git
   - Use HTTPS for webhooks
   - Verify signatures and API responses
   - Monitor for token expiration
